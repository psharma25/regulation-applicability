GHEC hardening controls — CIS v8 & CMMC L2

Terminology & how to use this

Each control below is collapsible by domain, tagged to CIS Controls v8 and CMMC L2, classified IG1–IG3, and carries a status tracker, a notes field, and a cost/dependency-risk indicator (green/amber/red dot). Status and notes are saved in this browser. Reference terms:

GHEC / GHES
GitHub Enterprise Cloud (GitHub-hosted SaaS, what the organization runs) vs Server (self-hosted appliance).
SSO
Single sign-on — log in once via the corporate identity provider instead of a separate password.
SAML / OIDC
The two federation protocols that carry a trusted login from the identity provider to GitHub.
Entra ID
Microsoft Entra ID (formerly Azure AD) — the organization's identity provider for SSO, MFA, SCIM, and Conditional Access.
SCIM
System for Cross-domain Identity Management — standard that auto-creates and removes accounts from the directory.
MFA / 2FA
Multi-factor / two-factor authentication — a second login factor beyond the password.
Conditional Access
Entra policy engine that enforces MFA and device posture before granting access.
PAT
Personal access token — an API/CLI credential; fine-grained PATs are scoped and expiring.
EMU
Enterprise Managed Users — GitHub identities fully owned and provisioned by the IdP; no personal accounts.
ZTNA
Zero-trust network access — identity-aware remote access replacing a flat VPN.
Dependabot
GitHub's built-in tool that alerts on vulnerable open-source dependencies and opens automated update PRs.
SCA
Software composition analysis — finding known vulnerabilities and licenses in third-party/open-source components.
SAST
Static application security testing — analyzing your own source code for vulnerabilities without running it.
CodeQL
GitHub's SAST engine that queries code as data to find security flaws.
GHAS
GitHub Advanced Security — now sold as two products: Secret Protection and Code Security (paid add-ons).
Push protection
Blocks a commit/push that contains a detected secret before it reaches the repository.
SBOM
Software bill of materials — a machine-readable inventory of every component in a build (SPDX/CycloneDX).
Attestation / SLSA
Signed build provenance proving where and how an artifact was produced (SLSA = Supply-chain Levels for Software Artifacts).
OIDC
OpenID Connect — here, lets a workflow get a short-lived cloud token instead of storing long-lived keys.
Runner
The machine that executes a GitHub Actions job. Self-hosted runners are your own hardware.
HIL
Hardware-in-the-loop — tests that drive physical equipment (industrial controllers, drives, robotic cells).
OT / IEC 62443
Operational technology and the industrial-control security standard governing its segmentation (Purdue model).
SIEM / Sentinel
Security information and event management; Microsoft Sentinel is the organization's SIEM for log retention and alerting.
CUI / DFARS
Controlled Unclassified Information; DFARS 252.204-7012 is the clause requiring its protection.
FedRAMP
US government cloud authorization program; Moderate is the level generally required for CUI.
CMMC L2
Cybersecurity Maturity Model Certification Level 2 — maps to NIST SP 800-171's 110 practices.
CIS Controls v8
Center for Internet Security's 18 prioritized safeguards, grouped into Implementation Groups IG1–IG3.
IG1 / IG2 / IG3
CIS Implementation Groups — a maturity ramp from essential hygiene (IG1) to advanced hardening (IG3).
PSIRT / CVD
Product Security Incident Response Team / Coordinated Vulnerability Disclosure — handling reported product flaws.
MISRA / CERT
Safety- and security-focused C/C++ coding standards Coverity enforces for embedded firmware.
800-171 families
Control groups: AC access control, IA identification/auth, AU audit, CM configuration, SC system/comms protection, SI system integrity, RA risk assessment, CA assessment, IR incident response.

Priority order

Set the time window for each Implementation Group. Entries are saved in this browser. IG1 is essential cyber hygiene, IG2 adds defense for sensitive data, IG3 hardens against advanced threats — sequence the control catalog accordingly.

IG1 Essential hygiene

  • SSO + SCIM (Entra ID), enforced MFA
  • Repository rulesets, signed commits
  • Dependabot + dependency graph
  • Secret Protection + push protection
  • Audit log streaming to Sentinel

IG2 Sensitive-data defense

  • Code Security / CodeQL + Autofix
  • Self-hosted HIL runner hardening, OIDC
  • Allowed-actions, environments, attestations
  • IP allow-list, custom roles & properties
  • Delegated bypass, Security Overview

IG3 Advanced hardening

  • Security campaigns at scale
  • Continuous-detection automation on the API
  • AI governance: content exclusions, FedRAMP models
  • EMU for CUI isolation
  • CMMC evidence / SSP continuous assurance

How CMMC L2, hardware-in-the-loop & a global workforce shape hardening

Architecture impact

CMMC L2 maps to NIST SP 800-171. Commercial GHEC carries only FedRAMP LI-SaaS (Low) — not sufficient for CUI under DFARS 252.204-7012. For CUI, the organization needs GitHub Enterprise Cloud for Government (FedRAMP Moderate) or GHEC with US data residency on the Moderate path. As SaaS, GitHub is an External Service Provider in scope — cite its ATO and document the shared-responsibility split.

What it hardens

CUI repos isolated and tagged via custom properties; EMU to prevent personal-account forks; restrict-to-FedRAMP-models + content exclusions for Copilot; stricter rulesets, delegated bypass, and Sentinel retention on CUI repos; non-CUI work in a separate org to limit assessment scope.

Watch-outs

Some CUI categories require FedRAMP High — confirm categorization first. Keep evidence of GitHub's authorization status current and map its FedRAMP controls to your 800-171 practices in the SSP.

Architecture impact

HIL jobs physically drive controllers, drives, and robotic cells, so they must run on self-hosted runners in the OT/lab network — even though the control plane is Cloud. That collides with IT/CMMC segmentation.

What it hardens

An isolated OT zone (IEC 62443 / Purdue) holds the HIL fleet and benches; runners are ephemeral, hardened, OIDC-authenticated, OT-labeled, and reach GitHub outbound-only with no inbound IT→OT path. AI Autofix/agent never auto-acts on device code.

Watch-outs

Runner labels must guarantee firmware jobs route only to the OT fleet. A poisoned action on an OT runner can reach a live machine — pin SHAs, allow-list actions, and segment the cell.

Architecture impact

With no network perimeter, a global workforce reaching the Cloud makes identity and data location the primary controls. Cloud helps: US/EU data residency is available natively, and Entra ID Conditional Access governs the front door.

What it hardens

Entra-enforced SSO + phishing-resistant MFA everywhere; SCIM offboarding so a leaver in any region loses access in minutes; IP allow-list + ZTNA at the edge; US data residency to keep code and Copilot inference in-region.

Watch-outs

Export control (ITAR/EAR) on defense-related industrial tech may restrict which foreign-national engineers can access which repos — enforce via EMU, repo isolation, and access reviews, not just network rules.

The overlap, in short: Both frameworks center on the same five pillars — identity & access, continuous vulnerability management, audit logging, secure configuration, and application software security — exactly where GitHub is strongest (CIS 5, 6, 7, 8, 16). The CIS Implementation Groups give a maturity ramp that aligns to a CMMC journey: IG1 covers baseline access, audit, and config practices many 800-171 controls require; IG2/IG3 add the depth and continuous detection assessors expect. GitHub does not address CIS 9 and 14 (browser/awareness) and only partially 10 and 11 (handled at the GHEC platform and endpoint layers). Use CIS v8 to drive engineering and CMMC L2 to frame the assessment.

Mappings to CIS Controls v8 and CMMC Level 2 / NIST SP 800-171 are illustrative engineering guidance — not a substitute for a C3PAO scoping determination. Whether GitHub falls inside the CUI boundary depends on the GHEC edition (commercial vs Government / data residency) and your contract's CUI categorization. Verify current GitHub feature availability, licensing, and FedRAMP status before committing architecture or spend.
